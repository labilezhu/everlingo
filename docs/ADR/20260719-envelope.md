## 1. 动机

### 1.1 即将到来的产品需求

下一阶段计划为产品加入一个新 feature：作为 Chrome Extension，为任意网页提供翻译与语言学习笔记功能。用户在浏览器选词 → 浮标 → sidePanel 打开聊天 → Agent 结合选词、上下文、网页 URL 完成 translate / look_up 任务。这个 feature 直接落地了 [PRODUCT-FUNC.md](../../PRODUCT-FUNC.md) 中"浏览器插件整合"卖点与 [docs/phases/phase2/product-phase-2.md](../phases/phase2/product-phase-2.md) 中"场景化地记下每一次查询"目标。

更进一步，未来还会有同族需求：
- PDF 阅读器插件：在任意 PDF 文档中选词查询
- EverLingo iOS app：在任意 app 内容中选词查询（类似 iOS 系统翻译集成）

这些"选词查询表面（selection surface）"的共同特征是：**Channel 在产出用户消息时，能同时提供结构化上下文（选词文本、上下文段落、来源 URL/文件、设备信息、用户期望任务）**。

### 1.2 现状的局限

当前 `Channel` 协议只产出纯文本：

```python
class Channel(ABC):
    async def recv(self) -> str | None: ...
```

存在以下问题：

1. **结构化上下文无处可放**：Chrome 选词翻译需要的 `selection / context / page_url / task` 等字段，无法在 channel → session → agent 链路中原生携带。
2. **来源差异被隐藏**：未来 PDF / iOS / web 等不同来源的相同字段（如 `selection.text`）需要不同语义，纯文本协议无法区分。
3. **无 schema 演进机制**：未来加字段（如截图、屏幕坐标）会是破坏性变更，无 `schema_version` 锁定。
4. **歧义**：如果直接在纯文本中嵌入 JSON，用户在 stdio 输入 `{"name":"mark"}` 会被 LLM 误判为结构化消息。

### 1.3 设计目标

- **一次定稳**：envelope schema 应能容纳当前已知的 web 场景 + 未来 PDF / iOS / epub 场景，避免协议频繁破坏性变更
- **Agent 不感知**：`MainAgent.ainvoke` 不应改签名接收 envelope 对象，渲染逻辑不应散落在多个组件
- **现有 channel 无损迁移**：stdio / wechat / web 现有行为应迁移为产 envelope，但不破坏现有用户体验
- **LLM 友好**：序列化格式应让 LLM 能精确引用字段（而非从 markdown 反向解析），并清晰区分结构化数据与用户自然语言输入

## 2. 合理性与可行性分析

### 2.1 合理性

**抽象正确**：所有"选词查询表面"都有共同的三件套——用户选了什么（selection）、周围是什么（context）、用户想说什么（chat.message）。再加 task（用户期望优先完成的任务）与 source（来源信息）即覆盖所有场景。这是产品本质抽象，不是为单一场景过度设计。

**task 作为 conversation mode 而非 RPC 命令**：与 [chat-agent-spec.md](../impl-spec/chat-agent-spec.md) 中"用户意图分析交由 LLM 判断"的设计哲学一致。`task` 仅作为 LLM 的先验偏好，不强制 Agent 行为。例如 `task=translate` 时用户追问"为什么这里用 bank"，Agent 应先翻译再解释。

**Agent 完全不感知 envelope 对象**：决策 4 选择"由 LLM 在调用 `request_memory_extraction` 时自己填 source 字段"，意味着 envelope 不需要透传到 Memory Writer。envelope 仅在 Session 层渲染为 LLM 可读文本后传给 `MainAgent.ainvoke(MessageEvent(text))`，Agent 签名不变。

**LLM 单一信源**：序列化为 `<envelope>{json}</envelope>` 后，LLM 看到的是带边界的 JSON 块，可以直接精确引用字段（如 `source.url`），不存在 markdown 渲染的 lossy 问题。

### 2.2 可行性

- **pydantic discriminated union**：`source` 字段用 `kind` 作为 discriminator，类型安全解析；未知 kind 默认 raise `ValidationError`，与决策"无 fallback"一致
- **现有 channel 全部可迁移**：
  - stdio/wechat：内部用 `wrap_plain_text(text)` 包装用户输入为最小 envelope（`task="none"`、`source.kind="plain"`）
  - web：HTTP body 改为 union，`{text}` 与 `{envelope}` 两种格式都支持；旧前端发 `{text}` 自动包装
- **测试改造可控**：grep 确认 `recv()` 调用点共 23 处，集中在 4 个测试文件 + 1 个 session.py + 4 个 channel 实现，全部列出（见 §4.6）
- **`MainAgent.ainvoke` 签名不变**：渲染后的文本仍是 `str`，通过 `MessageEvent(text=...)` 传入；现有 agent 单测零侵入

### 2.3 风险

| 风险 | 缓解 |
|---|---|
| LLM 输入格式从纯文本变成 `<envelope>{json}</envelope>`，可能影响现有 prompt 调优 | system prompt 新增"## 结构化用户输入（envelope）"节明确说明；跑完整 `test_main_agent.py` 验证不退化；如有退化调 prompt 措辞 |
| `test_session_event_queue.py` mock 改动较大（多个 `_make_session` 调用） | 逐个用例改 mock：`channel.recv` → `channel.recv_envelope`，断言 `ev.envelope.chat.message` |
| `web_acceptor.py` 的 `MessageBody` union 解析需 FastAPI 正确分发 | 加测试覆盖 `{text}` 和 `{envelope}` 两种 body 都能正确入队 |
| wechat/stdio channel 行为细微变化（多包了一层 envelope） | 不影响终端用户：渲染后 JSON 中 `chat.message` 字段即用户原话，LLM 仍能正确理解 |
| 未知 `source.kind` raise 阻塞 channel | 当前阶段不会发生（只有 `plain`）；未来若新增 kind 但旧 gateway 收到新 kind，行为明确（报错而非静默 fallback），便于发现协议不匹配 |

## 3. 架构变更内容

### 3.1 Channel 抽象变更

**变更前**：

```python
class Channel(ABC):
    @abstractmethod
    async def recv(self) -> str | None: ...
```

**变更后**：

```python
class Channel(ABC):
    @abstractmethod
    async def recv_envelope(self) -> UserInputEnvelope | None: ...
```

`recv()` **删除**，无 final 方法兜底。所有 channel 子类必须实现 `recv_envelope()`。Session 一律调 `recv_envelope()`。

### 3.2 数据流变更

**变更前**：

```
Channel.recv() -> str|None
  → Session._channel_listener 把 str 入 UserMessage.text
  → Session._handle_user_message(text) 把 text 传给 agent.ainvoke(MessageEvent(text))
```

**变更后**：

```
Channel.recv_envelope() -> UserInputEnvelope|None
  → Session._channel_listener 把 envelope 入 UserMessage.envelope
  → Session._handle_user_message(ev)
      → text = render_envelope_to_message_text(ev.envelope)
      → agent.ainvoke(MessageEvent(text=text))
```

### 3.3 组件变更

| 组件 | 变更 |
|---|---|
| `src/everlingo/gateway/channels/envelope.py` | **新增**：`UserInputEnvelope` pydantic 模型 + `wrap_plain_text()` + `render_envelope_to_message_text()` |
| `src/everlingo/gateway/channels/channel.py` | **删除** `recv()` 抽象方法；**新增** `recv_envelope()` 抽象方法 |
| `src/everlingo/gateway/channels/stdio_channel.py` | `recv()` → `recv_envelope()`，返回 `wrap_plain_text(line)` |
| `src/everlingo/gateway/channels/wechat_channel.py` | `recv()` → `recv_envelope()`，返回 `wrap_plain_text(msg.text)` |
| `src/everlingo/gateway/channels/web_channel.py` | `recv()` → `recv_envelope()`；`_incoming` 队列类型 `Queue[str\|None]` → `Queue[UserInputEnvelope\|None]` |
| `src/everlingo/gateway/web_acceptor.py` | `MessageBody` 改为 union（`{text}` 或 `{envelope}`）；`POST /message` 解析后统一转 envelope 入队 |
| `src/everlingo/gateway/session_events.py` | `UserMessage.text: str` → `UserMessage.envelope: UserInputEnvelope` |
| `src/everlingo/gateway/session.py` | `_channel_listener` 调 `recv_envelope()`；`_handle_user_message` 渲染 envelope + 日志改 envelope JSON 格式 |
| `src/everlingo/agents/agent.py` | `_build_system_prompt()` 在 `## 用户意图分类` 之前插入 `## 结构化用户输入（envelope）` 节 |
| `docs/impl-spec/envelope-spec.md` | **新增**：envelope 设计文档 |

### 3.4 不变

- `MainAgent.ainvoke(text: MessageEvent)` 签名不变，Agent 完全不感知 envelope 对象
- `request_memory_extraction` 工具 schema 不变
- `memory_extract_output_spec.md` 不变（等 Chrome Extension phase 再加 `source` 字段）
- Memory Writer Agent 不变
- vault 结构不变
- 三个 channel 的 `send()` / `send_typing_hint()` / `stop_typing_hint()` / `send_sound()` / `get_metadata()` 不变

## 4. 实现变更概述

### 4.1 `UserInputEnvelope` pydantic 模型

新文件 `src/everlingo/gateway/channels/envelope.py`：

```python
from __future__ import annotations
from typing import Annotated, Literal, Union
from pydantic import BaseModel, Field

TaskKind = Literal["translate", "look_up", "none"]


class ChatPart(BaseModel):
    message: str = ""


class SelectionPart(BaseModel):
    text: str = ""


class ContextPart(BaseModel):
    text: str = ""
    kind: Literal["paragraph", "page", "screen", "plain"] = "plain"


# —— Source tagged union —— 预留 5 个 kind，当前 channel 只产 plain ——
class SourcePlain(BaseModel):
    kind: Literal["plain"] = "plain"


class SourceWeb(BaseModel):
    kind: Literal["web"] = "web"
    url: str = ""
    title: str = ""


class SourcePdf(BaseModel):
    kind: Literal["pdf"] = "pdf"
    file_path: str = ""
    page_number: int | None = None


class SourceEpub(BaseModel):
    kind: Literal["epub"] = "epub"
    book_id: str = ""


class SourceIosApp(BaseModel):
    kind: Literal["ios_app"] = "ios_app"
    bundle_id: str = ""


SourcePart = Annotated[
    Union[SourcePlain, SourceWeb, SourcePdf, SourceEpub, SourceIosApp],
    Field(discriminator="kind")
]


class DevicePart(BaseModel):
    platform: Literal["chrome_ext", "ios_app", "pdf_reader", "web", "cli"] = "cli"
    locale: str | None = None
    timezone: str | None = None


class UserInputEnvelope(BaseModel):
    schema_version: int = 1
    task: TaskKind = "none"
    chat: ChatPart = ChatPart()
    selection: SelectionPart = SelectionPart()
    context: ContextPart = ContextPart()
    source: SourcePart = Field(default_factory=SourcePlain)
    device: DevicePart | None = None


def wrap_plain_text(text: str) -> UserInputEnvelope:
    """把纯文本输入包装为最小 envelope。stdio/wechat/web-{text} 请求用。"""
    return UserInputEnvelope(
        task="none",
        chat=ChatPart(message=text),
        source=SourcePlain(),
    )


def render_envelope_to_message_text(env: UserInputEnvelope) -> str:
    """把 envelope 序列化为 LLM 可读的 tagged JSON 字符串。

    Session._handle_user_message 在调用 agent.ainvoke 前用此函数。
    """
    json_str = env.model_dump_json(ensure_ascii=False)
    return f"<envelope>\n{json_str}\n</envelope>"
```

### 4.2 Channel 抽象

`src/everlingo/gateway/channels/channel.py`：

```python
class Channel(ABC):
    @abstractmethod
    async def recv_envelope(self) -> UserInputEnvelope | None:
        """读取一条用户输入；返回 None 表示 channel 结束。"""
        pass

    # send / send_typing_hint / stop_typing_hint / send_sound / get_metadata 不变
```

`recv()` **删除**，相关测试**删除**。

### 4.3 Channel 子类实现

`StdioChannel.recv_envelope()`：

```python
async def recv_envelope(self) -> UserInputEnvelope | None:
    try:
        line = (await asyncio.to_thread(input, "\n> ")).strip()
    except (EOFError, KeyboardInterrupt):
        print("\n再见！")
        return None
    if line.lower() == "/quit":
        print("再见！")
        return None
    if not line:
        return await self.recv_envelope()
    return wrap_plain_text(line)
```

`WechatChannel.recv_envelope()`：

```python
async def recv_envelope(self) -> UserInputEnvelope | None:
    text = await asyncio.to_thread(self._queue.get)
    if text is None:
        return None
    return wrap_plain_text(text)
```

`WebChannel.recv_envelope()`：保持原超时回收逻辑，仅 `_incoming` 队列类型改 `Queue[UserInputEnvelope | None]`，返回值不变（仍是 envelope 或 None）。

### 4.4 web_acceptor.py MessageBody union

class EnvelopeMessageBody(BaseModel):

# FastAPI 端点解析：有 envelope 字段 → 用 body.envelope；有 text 字段 → wrap_plain_text(body.text)

### 4.5 Session 与 session_events

`session_events.py`：

```python
@dataclass
class UserMessage(SessionEvent):
    """User input from channel.recv_envelope()."""
    envelope: UserInputEnvelope
```

`session.py`：

```python
async def _channel_listener(self) -> None:
    while True:
        env = await self.channel.recv_envelope()
        if env is None:
            logger.info("session %s: channel closed, posting QuitEvent", self.id)
            await self._event_queue.put(QuitEvent())
            return
        await self._event_queue.put(UserMessage(envelope=env))

async def _handle_user_message(self, ev: UserMessage) -> None:
    self.update_time = datetime.now()
    env_json = ev.envelope.model_dump_json(ensure_ascii=False)
    logger.debug(
        "[ChatAgent] IN session=%s channel=%s envelope=%s",
        self.id, self.channel_metadata.name, env_json,
    )
    text = render_envelope_to_message_text(ev.envelope)
    input_msg = MessageEvent(text=text)
    # ... 后续不变
```

### 4.6 system prompt 新节

在 `## 用户意图分类` 之前插入：

```markdown
## 结构化用户输入（envelope）
用户消息始终是 <envelope>...</envelope> 标签包裹的 JSON。字段说明：
- task: 用户偏好（translate/look_up/none），是偏好不是命令。即使 task=translate，用户仍可追问其他问题，你应正常回答
- chat.message: 用户的自然语言输入（可能为空，表示用户仅点击了 UI 按钮触发任务）
- selection.text: 用户选中的词/短语（可能为空，如纯聊天场景）
- context.text: 选词周围的上下文，用于消歧（如 bank 在河岸 vs 银行；可能为空）
- source: 来源信息（kind: web/pdf/epub/ios_app/plain；当前只有 plain，其他 kind 是预留）
- device: 设备信息（可能为 None）

标签内是 JSON，按 JSON 规则解析。
```

## 5. 决策

采用本方案。核心决策汇总：

| # | 决策点 | 选择 |
|---|---|---|
| 1 | Channel 抽象 | 删除 `recv()`，新增 `recv_envelope()` 抽象方法 |
| 2 | UserMessage 字段 | 单字段 `envelope: UserInputEnvelope`（不存渲染后 text） |
| 3 | 渲染时机 | `Session._handle_user_message` 按需调用 `render_envelope_to_message_text()` |
| 4 | MainAgent.ainvoke | 签名不变，仍收 `MessageEvent(text)`，Agent 完全不感知 envelope 对象 |
| 5 | 序列化格式 | `<envelope>{json}</envelope>` 标签包裹 JSON |
| 6 | source.kind 未知 | raise `ValidationError`（无 fallback） |
| 7 | memory_extract_output_spec | 不动，等 Chrome Extension phase |
| 8 | system prompt 节位置 | 在 `## 用户意图分类` 之前 |
| 9 | 设计文档 | 独立 `docs/impl-spec/envelope-spec.md` |
| 10 | task 枚举初始集合 | `translate / look_up / none` |
| 11 | source.kind 预留 | `plain / web / pdf / epub / ios_app`（5 值，当前仅 `plain` 实际使用） |
| 12 | web MessageBody | union（`{text}` 或 `{envelope}`），前端无感继续发 `{text}` |
| 13 | Channel.recv() 测试 | 一并删除，改为测 `recv_envelope()` |
| 14 | 日志格式 | `[ChatAgent] IN session=xxx channel=xxx envelope={"task":"none",...}` 直接打 envelope JSON |
| 15 | envelope 不含 UI 字段 | `rect` / `viewport` / 截图 等纯 UI 数据不在 envelope 中，留前端本地 |

## 6. 不在本次范围
